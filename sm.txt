    FEATURE: USER AUTHENTICATION & SESSION MANAGEMENT
    
    Purpose
    
    Handles user registration, login, logout, JWT-based session management, profile updates, and password changes. Acts as the gatekeeping layer for all protected routes.
    
    Backend Implementation
    
    Routes (/api/auth):
    - POST /register — Create account
    - POST /login — Authenticate
    - GET /me — Get current user
    - PATCH /profile — Update name/email
    - PUT /password — Change password
    - GET /users — List all users (Admin)
    - PATCH /users/:id/role — Update role (Admin)
    - DELETE /users/:id — Delete user (Admin)
    
    Controllers (authController.ts):
    - register: Validates fields, checks duplicate email, hashes password via Mongoose pre-save hook, generates JWT
    - login: Finds user by email with +password select, compares bcrypt hash, generates JWT
    - getMe: Returns sanitized user object from req.user (set by middleware)
    - getAllUsers: Returns all users sorted by createdAt descending
    - updateProfile: Partial update of name/email with duplicate email check
    - changePassword: Validates current password, saves new hashed password
    - updateUserRole: Admin-only role promotion/demotion
    - deleteUser: Admin-only user deletion
    
    Middleware (middleware/auth.ts):
    - protect: Extracts Bearer <token> from Authorization header, verifies JWT with JWT_SECRET, attaches req.user
    - authorize(...roles): RBAC gate checking req.user.role against allowed roles
    
    Models (models/User.ts):
    - Fields: name, email, password, role
    - role enum: 'user' | 'moderator' | 'admin' | 'ai_moderator' | 'expert'
    - password: select: false (not returned in normal queries), hashed via bcrypt salt factor 12 in pre-save hook
    - comparePassword: bcrypt.compare instance method
    - Indexes: email (unique)
    
    Database Collections:
    - yaksha_faq_users
    
    API Endpoints
    
    | Method | Path                     | Auth      | Notes                           |
    |--------|--------------------------|-----------|---------------------------------|
    | POST   | /api/auth/register       | Public    | Returns {token, user}           |
    | POST   | /api/auth/login          | Public    | Returns {token, user}           |
    | GET    | /api/auth/me             | JWT       | Returns {user}                  |
    | PATCH  | /api/auth/profile        | JWT       | Partial update                  |
    | PUT    | /api/auth/password       | JWT       | Requires current + new password |
    | GET    | /api/auth/users          | JWT+Admin | Paginated list                  |
    | PATCH  | /api/auth/users/:id/role | JWT+Admin | Role update                     |
    | DELETE | /api/auth/users/:id      | JWT+Admin | User deletion                   |
    
    Database Structure
    
    Users Collection (yaksha_faq_users):
    
    name: String (required, trimmed)
    email: String (required, unique, lowercase, regex validated)
    password: String (required, minlength 6, select: false, hashed bcrypt)
    role: String enum (default: 'user')
    createdAt: Date (auto)
    updatedAt: Date (auto)
    
    Indexes: email (unique)
    
    Frontend Implementation
    
    Pages:
    - LoginPage.tsx: Email+password form, navigates to / on success, shows error on failure
    - RegisterPage.tsx: Name+email+password form, validates 6-char password, navigates to / on success
    
    State Management:
    - AuthProvider (context) stores user, loading, isAuthenticated
    - Token stored in localStorage as yaksha_token, user as yaksha_user
    - On mount, if token exists, calls GET /auth/me to validate and hydrate user
    - On 401, clears storage and redirects to /login
    
    Hooks:
    - useAuth(): Returns {user, login, register, logout, loading, isAuthenticated}
    
    API Client (utils/api.ts):
    - Axios instance with JWT interceptor (reads yaksha_token from localStorage)
    - 401 response interceptor clears storage and redirects to /login
    
    User Experience
    
    1. User lands on /login, enters credentials
    2. useAuth().login(email, password) → POST /auth/login
    3. Token stored in localStorage, user hydrated into context
    4. ProtectedRoute checks isAuthenticated, redirects to / if logged in
    5. All API requests carry Authorization: Bearer <token>
    6. Token never expires until explicitly invalidated (no refresh mechanism)
    
    Admin/Moderator Functionality
    
    - Admin can list all users at GET /auth/users
    - Admin can update any user's role at PATCH /auth/users/:id/role
    - Admin can delete any user at DELETE /auth/users/:id
    - Moderator role cannot access user management endpoints (403 returned)
    
    Security Considerations
    
    - Passwords hashed with bcrypt (salt factor 12) before storage
    - JWT secret validated on every protected request
    - password field excluded from queries via select: false
    - No token refresh mechanism — tokens valid for 7 days (JWT_EXPIRES_IN ?? '7d')
    - Admin-only routes double-protected by both protect middleware and authorize('admin')
    - Email uniqueness enforced at DB level
    - Input validation present (name/email/password required, password minlength 6)
    
    Analytics/Logging
    
    - No authentication events are logged to the AdminLog collection currently
    - Search log captures query but not user identity
    
    Current Limitations
    
    - No token refresh — if token expires, user must re-login manually
    - No failed login attempt lockout (brute force vulnerable)
    - No 2FA support
    - No password reset / forgot password flow
    - No login throttling (rate limit applies per IP globally but not per account)
    - All logged-in users share same 7-day JWT expiry (no "remember me" option)
    
    Completion Status
    
    Fully Implemented — Core auth is complete. Missing password reset and token refresh are not critical for an internal FAQ portal but would be required for a public-facing production deployment.
    
    
    
    FEATURE: FAQ MANAGEMENT
    
    Purpose
    
    CRUD operations for Frequently Asked Questions, organized by category, with optional status workflow (pending/approved/rejected), helpfulness voting, and AI-powered FAQ matching for community posts.
    
    Backend Implementation
    
    Routes (/api/faq):
    - GET / — All FAQs grouped by category (or flat paginated)
    - GET /paginated — Flat paginated list
    - POST /check-match — AI FAQ matching against user query
    - GET /:id — Single FAQ
    - POST / — Create FAQ (Admin/Moderator)
    - PUT /:id — Update FAQ (Admin/Moderator)
    - DELETE /:id — Delete FAQ (Admin/Moderator)
    - PATCH /:id/feedback — Vote helpful/unhelpful (any logged-in user)
    
    Controllers (faqController.ts):
    - getAllFAQs: Returns {grouped} by default (category-grouped), or flat paginated {faqs, total, page, pages} when limit param is set
    - getPaginatedFAQs: Dedicated paginated endpoint with optional category filter
    - getFAQById: Single FAQ fetch (404 if not found)
    - createFAQ: Generates vector embedding on create (calls generateEmbedding()), stores in DB
    - updateFAQ: Recalculates embedding if question/answer/category changed
    - deleteFAQ: findByIdAndDelete
    - checkFAQMatch: Uses $vectorSearch against vector_index with threshold 0.82; returns matched FAQ or null
    - submitFeedback: Increments helpfulVotes or unhelpfulVotes counter
    
    Models (models/FAQ.ts):
    - Fields: question, answer, category, embedding, searchCount, status, views, helpfulVotes, unhelpfulVotes, createdBy
    - Text index on question + answer for keyword search
    - embedding field has select: false (hidden from normal queries)
    - Status enum: 'pending' | 'approved' | 'rejected'
    
    Database Collections:
    - yaksha_faq_faqs
    - Atlas vector index vector_index on embedding field (768-dim, cosine similarity)
    
    API Endpoints
    
    | Method | Path                  | Auth          | Notes                  |
    |--------|-----------------------|---------------|------------------------|
    | GET    | /api/faq              | JWT           | Grouped or paginated   |
    | GET    | /api/faq/paginated    | JWT           | Flat paginated         |
    | POST   | /api/faq/check-match  | JWT           | AI FAQ matching        |
    | GET    | /api/faq/:id          | JWT           | Single FAQ             |
    | POST   | /api/faq              | JWT+Mod/Admin | Create                 |
    | PUT    | /api/faq/:id          | JWT+Mod/Admin | Update                 |
    | DELETE | /api/faq/:id          | JWT+Mod/Admin | Delete                 |
    | PATCH  | /api/faq/:id/feedback | JWT           | Vote helpful/unhelpful |
    
    Database Structure
    
    FAQs Collection (yaksha_faq_faqs):
    
    question: String (required, trimmed)
    answer: String (required)
    category: String (required, trimmed)
    embedding: Number[] (768-dim, select: false)
    searchCount: Number (default: 0)
    status: String enum ['pending','approved','rejected'] (default: 'approved')
    views: Number (default: 0)
    helpfulVotes: Number (default: 0)
    unhelpfulVotes: Number (default: 0)
    createdBy: ObjectId (ref: User, default: null)
    createdAt: Date (auto)
    updatedAt: Date (auto)
    
    Indexes:
    - Text index on {question: 'text', answer: 'text'}
    - Vector index vector_index on embedding (managed in Atlas, not Mongoose)
    
    Frontend Implementation
    
    Pages:
    - FAQPage.tsx: Browse FAQs by category, search, sort (recent/relevant), paginated list, question detail view, related questions, confidence scoring
    
    Components:
    - CategoryGrid (category cards with top questions preview)
    - QuestionList (sortable, paginated list with load-more)
    - QuestionDetail (full question + answer with related items)
    - SearchBar (with suggest dropdown)
    - WordCloud (decorative category visualization)
    
    State: Component-level useState + useEffect fetching from GET /faq and GET /faq/paginated
    
    User Experience
    
    1. User navigates to /faq
    2. Sees category grid with top questions per category
    3. Can click category to see full question list
    4. Can search within FAQ via SearchBar (uses /search/suggest)
    5. Can sort list by "Most relevant" or "Most recent"
    6. Can click any question to see full answer
    7. Can vote helpful/unhelpful on FAQ
    
    Admin/Moderator Functionality
    
    - Moderator/Admin can create, edit, delete FAQs
    - FAQs have status: pending/approved/rejected
    - Admin panel (AdminFAQs.tsx) has full CRUD with approve/reject workflow
    - Failed queries from search analytics can be converted to FAQs directly
    
    Security Considerations
    
    - All FAQ routes require JWT (protect middleware)
    - Write operations require admin or moderator role via authorize()
    - embedding field not returned in normal queries (select: false)
    - Input validation: all required fields checked before DB write
    - No rate limiting specific to FAQ endpoints (uses global API limiter)
    
    Analytics/Logging
    
    - searchCount field exists on FAQ but is never incremented in current code — dead field
    - views field exists but is never incremented — dead field
    - helpfulVotes/unhelpfulVotes properly tracked
    - Admin actions on FAQs logged to AdminLog (approve/reject/edit/delete/create)
    
    Current Limitations
    
    - searchCount and views fields are never written to — purely stored
    - createdBy field is set on admin-created FAQs but the user's name is never populated when listing FAQs (only createdBy: ObjectId stored)
    - FAQ creation does not validate for duplicate questions
    - No soft delete / archive functionality
    
    Completion Status
    
    Fully Implemented — Core FAQ CRUD, AI matching, and voting are functional. Dead analytics fields (searchCount, views) are cosmetic issues, not functional gaps.

    - GET /:id — Single post

    FEATURE: HYBRID SEARCH (VECTOR + KEYWORD)
    
    Purpose
    
    Provides semantic (AI-powered vector) and keyword text search across both FAQs and community posts using Reciprocal Rank Fusion (RRF) to merge results from both rankers. Results cached in Redis (Upstash) with LRU fallback.
    
    Backend Implementation
    
    Routes (/api/search):
    - POST / — Execute hybrid search (JWT protected)
    - GET /trending — Get top 6 trending queries (JWT protected)
    - GET /suggest — Lightweight FAQ text suggestions (Public)
    
    Controllers (searchController.ts):
    - semanticSearch:
      1. Check Redis cache → return if hit
      2. Check LRU cache (process-local) → return if hit, promote to Redis
      3. Generate embedding via @xenova/transformers
      4. Run 4 searches in parallel: vector + text over both collections
      5. Apply source tags (faq vs community)
      6. Merge results via computeRRF() (RRF_K=60)
      7. Apply threshold filter (textScore > 0 OR vectorScore > 0.80)
      8. Cache in LRU + Redis
      9. Fire-and-forget: log to SearchLog
      10. Return top 5 results
    - getTrending: Aggregate SearchLog by normalized query, sort by count desc, limit 6
    - getSuggest: Regex search on question field, filters by status: 'approved', no auth required
    
    Utils (utils/search.ts):
    - computeRRF(vectorResults, textResults): Standard RRF with k=60, merges ranked lists
    - applySearchThreshold(results): Filters to textScore > 0 OR vectorScore > 0.80
    
    Utils (utils/embeddings.ts):
    - @xenova/transformers with Xenova/multi-qa-mpnet-base-dot-v1 (768-dim, mean-pooled, normalized)
    - Singleton cachedEmbedder initialized once, reused
    - warmEmbedder() pre-loads model (called by /api/warm endpoint)
    - generateEmbedding(text) → number[]
    
    Utils (utils/cache.ts):
    - Upstash Redis with lazy singleton initialization
    - Hash query with djb2-style hash → sc:<hash> cache key
    - TTL: 1 hour for search results
    - getCachedResults(query) → results or null
    - setCachedResults(query, results) → stores with TTL
    - invalidateCache() → deletes all result:* keys (not connected to any flow)
    
    API Endpoints
    
    | Method | Path                 | Auth   | Notes                            |
    |--------|----------------------|--------|----------------------------------|
    | POST   | /api/search          | JWT    | Hybrid search, returns 5 results |
    | GET    | /api/search/trending | JWT    | Top 6 trending queries           |
    | GET    | /api/search/suggest  | Public | FAQ text suggestions             |
    
    Database Structure
    
    Search Log Collection (yaksha_faq_searchlogs):
    
    query: String (required)
    resultsCount: Number (default: 0)
    topResultId: ObjectId (default: null)
    topResultSource: String enum ['faq','community',null] (default: null)
    createdAt: Date (auto)
    updatedAt: Date (auto)
    
    
    Frontend Implementation
    
    Pages:
    - HomePage.tsx: Main search interface with hero section, category grid, search bar with dropdown, results panel, trending queries, popular searches
    
    Components:
    - SearchBar: Controlled input, debounced, calls POST /search
    - SearchResults: Expandable result items with confidence tags (High/Medium), source badges
    - TrendingIssues: Shows trending search queries (from GET /search/trending)
    - CategoryGrid: Category pills with icons for quick search
    - TopSolved: Widget showing recently resolved community posts
    - CTA: "Need human answer?" prompt linking to community
    
    State: Component-level useState for query, results, trending, loading states
    
    User Experience
    
    1. User on home page types in SearchBar
    2. At 3+ characters, results panel appears with live search results
    3. Results show with source badge (FAQ/Community), category, confidence tag
    4. Can expand result to see full answer
    5. Left panel shows matching questions, right panel shows suggestions and trending
    6. Can click category pill to search that category
    7. If no results, shows "No matches found. Try a different phrase or ask the community."
    8. Popular searches shown below suggestions
    
    Admin/Moderator Functionality
    
    - AdminSearch.tsx: Shows top search keywords, search activity over time, failed query heatmap
    - AdminAnalytics.tsx: Shows failedQueries with "Add to FAQ" button to convert failed query to FAQ
    - Search insights used to identify content gaps
    
    Security Considerations
    
    - /search requires JWT (protected)
    - /search/suggest is public — no auth required (only returns FAQ titles/categories)
    - Rate limiting: 300 req/15min per IP via global apiLimiter
    - No query injection possible in getSuggest (regex chars escaped)
    - No user identity stored with search logs
    
    Analytics/Logging
    
    - Every POST /search logs: query, resultsCount, topResultId, topResultSource, timestamp
    - getTrending reads from SearchLog aggregation
    - AdminAnalytics shows failed queries (resultsCount=0)
    - No real-time analytics — all aggregate-based
    - trending queries endpoint is unauthenticated — public
    
    Current Limitations
    
    - Search log has no index — aggregation queries on large datasets will be slow
    - invalidateCache() exists but is never called — cache only expires via TTL
    - No search analytics by user — only aggregate counts
    - No A/B testing or result click tracking
    - Upstash cache is per-deployment — different Vercel instances don't share cache unless using same Upstash instance (which is configured via env vars, so they do share it — but the in-process LRU cache is per-instance)
    
    Completion Status
    
    Fully Implemented — Hybrid search, caching, trending, suggest are all functional. Missing click tracking and result relevance feedback are nice-to-haves.
    
    
    
    FEATURE: ADMIN DASHBOARD & ANALYTICS
    
    Purpose
    
    Comprehensive admin panel for managing FAQs, community posts, users, viewing system analytics, search insights, reports, and activity feed.
    
    Backend Implementation

    - GET /faq-growth — FAQ growth over time
    - GET /top-categories — Category breakdown
    - GET /search-insights — Search analytics
    - GET /users — User list (paginated, searchable)
    - GET /faqs — FAQ list (paginated, filterable)
    - GET /reports — Reports data (FAQs + SearchLogs)
    - GET /activity-feed — Admin action logs
    - GET /user-activity-chart — User/search activity chart
    - GET /community/posts — Community post list
    - POST /faq — Create FAQ via admin
    - POST /faq/approve — Approve FAQ
    - POST /faq/reject — Reject FAQ
    - PUT /faq/:id — Update FAQ
    - DELETE /faq/:id — Delete FAQ
    - DELETE /community/:id — Delete community post

    - getFaqGrowth: Groups FAQs by day over specified period, fills missing days with 0

    Routes (/api/analytics):
    - GET / — Aggregate search analytics (total, popular queries, failed queries)
    - GET /failed-queries — Top 30 failed queries from last 7 days
    
    Controllers (analyticsController.ts):

    API Endpoints
    
    See above table. All /api/admin/* routes require adminOnly middleware (verifies JWT + admin/moderator role).
    
    Database Structure
    
    AdminLog Collection (yaksha_faq_adminlogs):
    
    adminId: ObjectId (ref: User, required)

    FEATURE: EMBEDDINGS & VECTOR SEARCH INFRASTRUCTURE
    
    Purpose
    
    Provides the AI embedding pipeline for semantic search — loads a transformer model, generates 768-dim vectors for FAQ and community post content, and stores them in MongoDB Atlas for vector search.
    
    Backend Implementation
    
    Utils (utils/embeddings.ts):
    - Uses @xenova/transformers (WebAssembly-based, no server GPU needed)
    - Model: Xenova/multi-qa-mpnet-base-dot-v1 (768-dim, dot-product similarity)
    - Singleton pattern: cachedEmbedder initialized once, reused
    - generateEmbedding(text): Returns number[] via mean pooling + normalization
    - warmEmbedder(): Pre-loads model (called by POST /api/warm)
    
    Usage in controllers:
    - faqController.createFAQ: Generates embedding on create
    - faqController.updateFAQ: Regenerates embedding if content changed
    - communityController.createPost: Generates embedding (non-critical failure is caught)
    - searchController.semanticSearch: Generates query embedding for vector search
    - communitySearchController.searchCommunityPosts: Generates query embedding
    
    Database:
    - Atlas vector index vector_index on yaksha_faq_faqs.embedding and yaksha_faq_communityposts.embedding
    - 768 dimensions, cosine similarity
    - numCandidates: 50 (FAQ match), numCandidates: 100 (community search)
    
    Frontend Implementation
    
    No direct frontend interaction with embeddings. Frontend consumes search results which are powered by vector search.
    
    Current Limitations
    
    1. No auto-embedding on community post update: updatePost does not regenerate embedding when post is edited
    2. No auto-embedding on FAQ bulk import: If an FAQ is updated via admin panel without going through the updateFAQ controller, embedding won't be regenerated
    3. Cold start: Model is lazy-loaded on first generateEmbedding call — first search request is slow. /api/warm exists but isn't automatically called in production
    4. No model versioning: If the model needs to change, all existing embeddings must be regenerated via backfill:embeddings script
    
    Completion Status
    
    Fully Implemented — Core embedding pipeline works. Cold start and update regeneration are operational concerns, not missing features.
    
    
    
    FEATURE: CACHING LAYER (REDIS + LRU)
    
    Purpose
    
    Reduce redundant AI embedding generation and search computation by caching results. Uses Upstash Redis (serverless-compatible) as shared cache, with in-process LRU as fallback.
    
    Backend Implementation
    
    Utils (utils/cache.ts):
    - Lazy singleton Redis client (only initializes if REDIS_URL + REDIS_TOKEN set)
    - Query hashing: djb2-style string hash → sc:<hash> cache key
    - Cache TTL: 1 hour (RESULT_TTL = 60 * 60)
    - getCachedResults(query): Redis GET, returns null on miss
    - setCachedResults(query, results): Redis SET with TTL
    - invalidateCache(): Deletes all result:* keys
    
    Integration in searchController.ts:
    1. Check Redis cache (shared across all serverless instances)
    2. Check LRU cache (process-local)
    3. On LRU hit, promote to Redis for cross-instance benefit
    4. On miss, compute and store in both LRU + Redis
    
    Frontend Implementation
    
    No frontend interaction with cache directly. Frontend sees faster search results due to caching.
    
    Current Limitations
    
    1. invalidateCache() is never called: Cache invalidation function exists but no code path triggers it. When FAQ content changes, stale results may be served until TTL expiry.
    2. No cache warming on deployment: No mechanism to pre-warm cache with popular queries
    3. Redis failure is silently swallowed: If Redis is unavailable, cache just returns null — no alerting
    4. No cache statistics: Hit/miss rates not exposed or logged in a structured way
    
    Completion Status
    
    Fully Implemented — Cache layer is functional. Missing invalidation trigger and monitoring are operational gaps.

    MISSING PRODUCTION FEATURES
    
    P0 Critical
    
    1. SearchLog has no database index — severe performance risk at scale
    
    searchLogSchema has no indexes. Every analytics query (popular queries, failed queries, trending, activity) does a full collection scan with $group. At 100K+ search logs, every admin page load will be slow.
    
    Recommended: Add compound index {createdAt: -1} and text index or hashed index on query for aggregation performance.
    
    
    
    2. Notification delivery mechanism is entirely absent
    
    Settings are stored, notification documents are created for some events, but no email, push, or in-app notification is actually delivered based on user preferences. Users have no way to receive urgent notifications like "your post was resolved" without polling.
    
    Recommended: Integrate a notification delivery system (e.g., email via SendGrid/Resend, or in-app polling with WebSocket for real-time).
    
    
    
    3. No cache invalidation on FAQ/community update
    
    Stale search results cached for up to 1 hour after content changes. A newly created FAQ won't appear in search results for up to 1 hour.
    
    Recommended: Call invalidateCache() from FAQ and community post CRUD operations.
    
    
    
    P1 Important
    
    4. No token refresh — sessions expire with no renewal
    
    JWT tokens are valid for 7 days with no refresh mechanism. If a token expires, the user must explicitly re-login. For an internal tool this is acceptable; for a production public-facing app this causes user experience issues.
    
    Recommended: Implement refresh token rotation or extend JWT expiry for active users.
    
    
    
    5. No rate limiting per account (only per IP)
    
    Global rate limiter is per-IP. A single user with a valid token can exhaust the 300 req/15min budget and affect other users on the same IP.
    
    Recommended: Implement per-user rate limiting using the authenticated user's ID.
    
    
    
    6. No duplicate question detection on FAQ creation
    
    Creating two FAQs with identical or very similar questions is possible. No similarity check is run before saving.
    
    Recommended: Run a vector search similarity check before FAQ creation; warn or block if similarity > 0.9.
    
    
    
    7. User activity chart uses fake data
    
    Admin dashboard shows completely random user counts in the activity chart, giving a false picture of system usage.

    ARCHITECTURAL ISSUES
    
    Severity: HIGH
    
    Issue 1: SearchLog has no indexes — full collection scans on every admin query
    
    - Why it matters: Every admin analytics page load executes aggregation pipelines on SearchLog without any index support. At 100K+ records, these queries will take 5-10+ seconds, blocking the admin panel.
    - Affected queries: getTrending, getSearchAnalytics, getFailedQueries, getSearchInsights, getUserActivityChart, getFaqGrowth
    - Recommended solution: Add {createdAt: -1} index on SearchLog. For query text aggregations, a hashed index on query or a compound {query: 1, createdAt: -1} would speed up the $group stage.
    
    
    
    Issue 2: No cache invalidation — stale results for up to 1 hour
    
    - Why it matters: User searches may return outdated results after FAQ or community content changes. A newly added FAQ won't appear in search for 1 hour.
    - Recommended solution: Call invalidateCache() from all FAQ and community post write operations (create, update, delete).
    
    
    
    Issue 3: Upstash Redis credentials stored in plain .env — git-tracked potentially
    
    - Why it matters: REDIS_URL and REDIS_TOKEN are in .env which may be committed to git history.
    - Recommended solution: Use environment variables injected at deploy time (Vercel Environment Variables) rather than .env file.
    
    
    
    Severity: MEDIUM
    
    Issue 4: getUserActivityChart returns fake user count data
    
    - Why it matters: The admin dashboard displays a user activity chart that shows random numbers for the users metric, providing no actual insight into user engagement.
    - Recommended solution: Track actual unique user activity per day in a separate collection or use an existing event log to aggregate.
    
    
    
    Issue 5: Two separate Admin implementations — confusing and duplicated
    
    - Why it matters: There's a legacy AdminPage.tsx (at /admin/old) and a new AdminLayout-based admin panel with separate pages. Both access the same API endpoints. The legacy page has its own analytics, FAQ, community, and user tabs. The new admin panel mirrors all of these with different UI.
    - Recommended solution: Deprecate and remove the legacy /admin/old route and AdminPage.tsx. The new admin panel at /admin is comprehensive.
    
    
    
    Issue 6: Admin activity feed has no pagination — only 20 records
    
    - Why it matters: getActivityFeed returns only 20 most recent entries with no pagination. Audit trail is limited to recent activity.
    - Recommended solution: Add pagination with ?page=&limit= to getActivityFeed.
    
    
    
    Issue 7: No hard delete cleanup for orphaned data
    
    - Why it matters: Deleting a user doesn't delete their posts, comments, or notifications. Deleted users' names will show as "Student" or "User" in UI due to population failing, but data remains in DB.
    - Recommended solution: Add userId cascade delete or set author to null with a [deleted] placeholder on user deletion.
    
    
    
    Issue 8: Notification settings are stored but never read — dead preference system
    
    - Why it matters: Users configure notification preferences that have zero effect on what notifications they receive. The settings are completely inert.
    - Recommended solution: Either remove the settings UI (simplifies) or implement a preference check before sending any notification.
    
    
    
    Severity: LOW
    
    Issue 9: No structured logging — console.log scattered
    
    - Why it matters: The logger utility exists but cache operations use console.log directly. Search operations have no structured logging. Makes debugging in production harder.
    - Recommended solution: Use logger.info/warn/error consistently across all utilities.
    
    
    
    Issue 10: No health check for Redis in /api/health
    
    - Why it matters: /api/health only pings MongoDB. If Redis is down, the health check returns ok even though search caching is degraded. No way to know from health endpoint.
    - Recommended solution: Add Redis ping to health check and include cache status in response.
    
    
    
    Issue 11: Moderate use of any type casting in community controller
    
    - Why it matters: (post.comments as any).id(req.params.commentId) bypasses TypeScript safety. If the Mongoose subdocument API changes, this silently breaks.
    - Recommended solution: Use proper typed subdocument accessor or cast through a more specific type.
